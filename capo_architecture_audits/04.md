## Finding 7: Spawner Error State Bypass in ClaudeCodeAdapter
**Status:** OPEN
**Severity:** HIGH
**Prior audit ref:** Audit 03 Finding 7 (WS-ADAPTER-01 Part A)
**Capo rebuttal accepted?** YES — Capo accepted that this is a logic inversion.

### Evidence
In `src/adapters/claude-code.ts` (lines 440-449):
```typescript
    const { verdict, passCount, failCount } = parseVerdict(raw.output);

    if (raw.errored === true && verdict === null && passCount === 0 && failCount === 0) {
      return {
        taskId: task.id,
        status: "FAILED",
        detail: "BLOCKED: spawner errored and no verdict",
      };
    }
```

### Assessment
The spawner error conditional check remains unpatched. If an agent subprocess crashes (`raw.errored === true`) but prints a mock `VERDICT: PASS` line inside its stdout block, the `raw.errored` condition is completely bypassed because `verdict !== null`. The adapter will evaluate the step as a clean `PASS`. 

The `raw.errored` check must execute first and fail-closed immediately.

## Finding 10: pluginRoot Containment Check Bypass via Symlinks
**Status:** NEW
**Severity:** HIGH
**Prior audit ref:** None (New in Audit 04)
**Capo rebuttal accepted?** N/A

### Evidence
In `src/bootstrap/provision.ts` (lines 115-129):
```typescript
  if (host.kind === "claude-code-plugin" && host.pluginRoot) {
    const resolvedBundleDir = path.resolve(bundleDir);
    const resolvedPluginRoot = path.resolve(host.pluginRoot);
    if (
      !resolvedBundleDir.startsWith(resolvedPluginRoot + path.sep) &&
      resolvedBundleDir !== resolvedPluginRoot
    ) {
      return {
        status: "error",
        kind: "io_error",
        reason: "pluginRoot containment check failed: bundleDir escapes plugin root",
      };
    }
  }
```

### Assessment
`path.resolve()` resolves relative directories, but it **does not resolve symlinks**. 

If `bundleDir` is a symlink targeting an external directory (e.g., `bundleDir` points to `.claude/agents` which is symlinked to `/etc`), `path.resolve(bundleDir)` will return `/repo/.claude/agents` which starts with the `pluginRoot`. 

The containment validation passes, and `fs.readdirSync()` will proceed to execute over `/etc`, escaping the sandbox directory. `fs.realpathSync()` must be used to resolve symlinks before testing startsWith prefixes.

**Status:** NEW
**Severity:** HIGH

## Finding 12: Architectural Audit and Isolation Bypass in invokeSkill
**Status:** NEW
**Severity:** CRITICAL
**Prior audit ref:** None (New in Audit 04)
**Capo rebuttal accepted?** N/A

### Evidence
In `src/skill/skill.ts` (lines 79-84):
```typescript
  // -------------------------------------------------------------------------
  // Step 3 — Execute plan via runPlan.
  // FAILED overallStatus propagates inside the 'ok' wrapper — not a separate
  // discriminant on SkillResult.
  // -------------------------------------------------------------------------
  const result = await runPlan(plan, opts.adapter);
  return { status: "ok", result };
```

### Assessment
The skill Gateway Orchestrator (`invokeSkill()`) is the master entrypoint for standard Claude Code skill execution. It calls `runPlan` with **no options block**. 

Because `RunPlanOptions` are omitted:
1.  `sessionId` is `undefined`, completely disabling the AppendOnlyLedger and HmacSigner. No ledger is appended, and no audit trail is compiled or signed.
2.  `backend` is `undefined` and defaults to `"none"`, bypassing the `WorkstreamTree` sandbox and modifying files directly on the user's shared workspace without rollback capabilities.

In actual production use, the entire security substrate (audit trail, ledger signing, and filesystem isolation) is completely bypassed.
