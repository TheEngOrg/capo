# ADR-072: Direct Task() Dispatch — Relay Model Removal

## Status

Accepted

## Context

Capo shipped with a `GATEWAY_SPAWN_REQUEST` relay protocol embedded in its agent definition. Under this model, Capo did not call `Task()` directly. Instead, it emitted a delimiter-fenced markdown block:

```
GATEWAY_SPAWN_REQUEST
subagent_type: <role>
model: <model-id>
prompt: ...
END_GATEWAY_SPAWN_REQUEST
```

The main Claude Code session — labeled "Dispatcher" in docs and agent files — was supposed to parse this block and execute `Task()` on Capo's behalf, then relay the output back verbatim in the next Capo turn.

This model was never the intended design. Capo has `Task` in its tools list. The relay was an over-interpretation of an early "main session = gateway" framing that was meant as a routing note, not an architectural two-tier constraint.

The consequences were significant:

- Capo never called `Task()` directly, which meant all ~36 prior workstreams ran outside the deterministic engine. The engine's gates, signed ledger, and plan validation never executed on the actual plugin path.
- A full session of anomalous behavior on 2026-06-24 traced directly to this framing: the main session was acting as a relay proxy, introducing latency, interpretation errors, and an extra layer of prompt rewriting that corrupted specialist instructions.
- The "Dispatcher" identity label was embedded in `agents/capo.md`, `docs/how-it-works.md`, and `sandbox/README.md`, causing ongoing drift as new contributors and agent prompts read the incorrect architecture from authoritative docs.

## Decision

Remove the relay model entirely.

1. **Capo calls Task() directly** for all specialist dispatches. The fenced `GATEWAY_SPAWN_REQUEST` block format is removed from `agents/capo.md`.
2. **Rotation spawns are also direct Task() calls** — `Task()` with `subagent_type: "teo:capo"` and a rotation prompt containing the checkpoint path.
3. **The "Dispatcher" identity label is retired** from all agent definitions, docs, and sandbox guides.
4. **The main session is a thin gateway** — it invokes the `/teo` skill; Capo handles all orchestration from that point forward.

Files changed:
- `agents/capo.md` — relay section removed; direct-dispatch section authored; rotation steps updated; "Dispatcher" sentence removed
- `.claude/agents/capo.md` — mirrored to match `agents/capo.md` (byte-for-byte)
- `docs/how-it-works.md` — rewritten to show `You → /teo → Capo → specialists` flow
- `sandbox/README.md` — STEP-3A updated; "Dispatcher CLAUDE.md" reference removed

## Consequences

**Positive:**
- Capo's `Task()` calls are direct and auditable — no proxy layer to introduce rewriting or latency.
- Rotation spawns use the same `Task()` mechanism as every other specialist dispatch — no special-case protocol.
- The relay bypass is closed: future wiring of `Task()` calls through the runtime engine (gates, signed ledger) is now possible. All ~36 prior workstreams bypassed the engine; this ADR removes the structural reason they had to.
- Docs and agent files now describe the actual architecture, reducing drift for future contributors and agent prompts.

**Neutral:**
- Existing workstream history ran under the relay model. No retroactive change is made to prior run traces or ledger entries.
- The `relay_authorization` field in `capo.md's` `directive_gate` block is not removed in this change — it guards against unauthorized relayed instructions and remains a useful safety constraint even without the relay model.
