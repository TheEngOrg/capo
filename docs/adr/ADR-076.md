# ADR-076: HMAC Signing Wired into Gate Evaluation Path

## Status

Proposed

## Context

Gate evaluation (`evaluate-gate`) currently produces plain JSON output with a verdict, gate metadata, and a ledger entry. There's no cryptographic proof that the gate was actually checked — the output is just a file we wrote ourselves, readable but not verifiable.

For the YC demo, we need to show that the gate system is real. "Trust us, it ran" isn't a demo. We need round-trip verifiable evidence: a signed gate result that a second command can independently verify against the same key, plus a structured telemetry artifact that a web GUI can poll live during the demo.

The signing primitives already exist. `HmacSigner` (HMAC-SHA-256) and `AppendOnlyLedger` are in `src/core/sign.ts` and `src/core/ledger.ts` — tested, production-grade, with pipe-injection defense and constant-time comparison. This ADR wires them into the gate evaluation path.

## Decision

### D1 — plan_id source: EXECUTION_PLAN artifact ID

The `plan_id` field in the `SignPayload` comes from the EXECUTION_PLAN artifact ID — not the gate_id.

The execution plan already has a stable identifier generated at plan initialization (`plan-init` command output). That ID flows as an explicit argument into `evaluate-gate`, which then uses it as `plan_id` in the `SignPayload`. This binds the signature to the actual execution plan, not to an individual gate check. It's the right semantic: you're proving this gate result belongs to this plan.

The gate_id is still included in the output — it's just not the signing anchor.

### D2 — Full 64-char hex gate_sig in the STEP_ARTIFACT block

`evaluate-gate` includes a `gate_sig` field in its JSON output alongside the existing gate fields. The value is the full 64-character lowercase hex HMAC-SHA-256 signature produced by `HmacSigner.sign()`.

No truncation. The full sig goes in the output block so `verify-gate` has exactly what it needs to re-derive and compare.

### D3 — `verify-gate` command + structured telemetry artifact

Two outputs from a signed gate evaluation:

**`verify-gate` command (~30 lines):** Calls `HmacSigner.verify()` against the gate_sig from the STEP_ARTIFACT block. Returns a clear pass/fail with the verified fields printed. This is the human-facing round-trip proof for the demo — run `evaluate-gate`, then run `verify-gate` on the output, watch it pass.

**Telemetry artifact:** The signed gate result — gate_sig, ledger entries, verdict, and gate metadata — lands in a structured JSON file at a predictable path:

```
<ledger_base_dir>/telemetry/<session_id>-gate-results.json
```

This path is machine-readable and stable within a session. A web GUI polling this file during the demo gets live gate results without any server. The file is written atomically (write to temp, rename) to avoid partial reads.

### D4 — Same-machine only; key at `~/.teo/keyring/default.key`

Sign and verify on the same laptop. The HMAC key lives at `~/.teo/keyring/default.key` — the existing `HmacSigner` default path. It's auto-generated on first use (32 cryptographically-random bytes, mode 0600, directory mode 0700). No key distribution, no cross-machine verifiability.

This is deliberate scope containment for the demo. HMAC isn't the right primitive for cross-machine audit without key distribution infrastructure we don't have yet. Same-machine verification is sufficient to prove the gate path is real and the signatures are live.

## Implementation Pieces

**1. `evaluate-gate` wiring**

Add `plan_id` as an explicit CLI argument. After the gate evaluation logic runs, construct a `SignPayload`:

```
plan_id: <from CLI arg>
task_id: <gate_id> (the gate being evaluated is the task-scoped event)
actor_id: "evaluate-gate"
verdict: <PASS | FAIL | BLOCKED>
ts: <ledger-assigned>
seq: <ledger-assigned>
content_hash: null (no target_dir for gate checks)
```

Call `HmacSigner.sign()`, append `gate_sig` to the STEP_ARTIFACT JSON output, write the telemetry artifact.

**2. `verify-gate` command**

Reads the STEP_ARTIFACT JSON (from file or stdin), reconstructs the `SignPayload` from the fields in the artifact, calls `HmacSigner.verify()`. Exits 0 on valid, non-zero on tampered/invalid. Output is human-readable for the demo and machine-readable (JSON mode flag) for scripting.

**3. Telemetry artifact**

Written to `<ledger_base_dir>/telemetry/<session_id>-gate-results.json` (defaults to `~/.teo/telemetry/`). The base dir is injectable for tests — telemetry must use the same `baseDir` resolution as `HmacSigner` and the ledger. Accumulates gate results across a session (array, appended per gate). Structure:

```json
{
  "session_id": "...",
  "gates": [
    {
      "gate_id": "...",
      "plan_id": "...",
      "verdict": "PASS",
      "gate_sig": "<64 hex chars>",
      "ts": "...",
      "seq": 1
    }
  ]
}
```

## Consequences

**Positive:**
- Gate evaluation is now cryptographically authenticated: a second command can verify the output was produced by a process holding the same key. It doesn't prove the gate logic is correct, but it proves the output wasn't composed ad-hoc after the fact. Demo-able with a single follow-up command.
- The `gate_sig` binding to `plan_id` (EXECUTION_PLAN artifact ID) ties the proof to the actual plan — not an arbitrary string we made up at gate time.
- The telemetry artifact gives a web GUI a stable polling target with no server required. Live gate results during the demo without any infrastructure.
- `HmacSigner.verify()` uses `crypto.timingSafeEqual` — constant-time comparison, pipe-injection defense already tested. We're not adding new security primitives, just wiring existing ones.
- `verify-gate` is ~30 lines. Small surface, easy to audit.

**Negative:**
- Same-machine only. The HMAC key never leaves `~/.teo/keyring/`. Cross-machine audit would require key distribution or a different signing scheme (asymmetric, HSM). Out of scope for now, but this is the ceiling on the guarantee.
- The key at `~/.teo/keyring/default.key` is a 32-byte binary file in plaintext. It is not stored in the OS keychain or any encrypted secret store. File mode 0600 limits read access to the owner process, but root or any process running as the same user can read it. A compromised user account = compromised signing key = forged signatures are undetectable.
- `plan_id` must flow explicitly as a CLI argument into `evaluate-gate`. If a caller forgets to pass it (e.g. a script that constructs the command), the signing either fails or uses a fallback — the command must enforce it as a required arg, not optional.
- The telemetry artifact accumulates for the session lifetime. No rotation or pruning built in for v1. Fine for demo; needs a cleanup story before long-running production use.
- Signing depends on the ledger assigning `ts` and `seq` before the signature is computed. If the ledger write fails (disk full, concurrent write error), `evaluate-gate` must not emit an unsigned STEP_ARTIFACT silently — it should exit non-zero. The failure mode needs to be explicit in the implementation, not defaulted to 'partial output.'
