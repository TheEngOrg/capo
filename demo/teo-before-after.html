The core shift

LLM ran on everything. Now it runs in two places.

In v4.x the LLM touched every decision point — routing, verification, role assertions, task dispatch. That's expensive, slow, and non-deterministic. TEO 5 collapses that to exactly two call sites.

Before — v4.4.4
LLM involved at every decision point

Routing, verification, identity assertions, task dispatch — all passed through an LLM call. Each step carried hallucination risk and token cost.

Every red dot = an LLM call

After — TEO 5.0.0
LLM in exactly 2 places

Sage plans (1 LLM call). Each AGENT task spawns one call. SCRIPT tasks run at zero tokens. Everything else is deterministic code with exit codes.

The recurring issues

Six things we kept hitting

These weren't theoretical. Each one either produced a real false positive, left an enforcement gap in production, or silently raised costs.

01

Stale-token replay

v4.x — The symptom

A token file left in .claude/memory/ got replayed against a new session and falsely asserted role=sage. A sign-off from a previous run looked valid in a completely different execution context.

# Token written once, replayed indefinitely identity-tokens/sage-token-2024-11.json # session_id not bound to spawn # no sequence number, no expiry
TEO 5 — The fix

Every verdict is HMAC-SHA256 over plan_id | task_id | actor_id | verdict | ts | seq. The seq and ts fields are inside the signed message — a captured sign-off can't be replayed because the sequence number won't match the current context.

HMAC-SHA256( plan_id | task_id actor_id | verdict ts | seq # ← inside the signed blob )
02

Token forgery

v4.x — The symptom

The identity token was plain base64url JSON. The issuance script itself included a warning: "Current format (base64url JSON) is NOT cryptographically non-forgeable." Anyone with write access to the directory could produce a valid-looking token.

# WARNING (in the script itself): # base64url JSON is NOT # cryptographically non-forgeable. echo $payload | base64url > token.json
TEO 5 — The fix

Verdicts are HMAC-signed with a key stored in ~/.teo/keyring/ at mode 0600. The key is never checked into any repository. A forged verdict without the key fails verification on the first gate check.

~/.teo/keyring/signing.key 0600 # never in .claude/, never in repo # HMAC verification at every gate
03

Fail-open gates

v4.x — The symptom

Audited flows ran "0 BLOCK, N WARN." Advisory gates were documented as target: block but stayed at WARN. One validator (teo-bash-arg-validate.sh) was silently fail-open for weeks, leaving a hardening rule unenforced in production.

teo-bash-arg-validate.sh # exit 0 on any input ← fail-open # "target: block" ← never enforced RESULT: 0 BLOCK, 14 WARN
TEO 5 — The fix

Mechanical verification is an exit code — pass=0, anything else blocks. No "advisory" mode in the deterministic core. Gate verdicts are signed, so a gate that passes without producing a valid signature fails at the next checkpoint.

# exit 0 → pass (signed) # exit !0 → block (no sign-off issued) # no WARN-only path in core gates
04

Memory parasitism

v4.x — The symptom

Run-state lived in .claude/memory/ alongside config. Stale identity-tokens/, go-signals/, and _v4-disabled-traces/ directories accumulated. That noise drove enforcement drift — it wasn't clear what was authoritative.

.claude/memory/ identity-tokens/ ← stale go-signals/ ← stale _v4-disabled-traces/ ← stale plans/ ← live?
TEO 5 — The fix

All run-state moves to ~/.teo/. .claude/ keeps only what Claude Code needs to route. The boundary rule is simple: "if losing it corrupts an audit or finance trail → ~/.teo/; if Claude Code needs it to route → .claude/."

~/.teo/ # audit, keys, run-state .claude/agents/ # routing only .claude/skills/ # routing only
05

LLM on mechanical work

v4.x — The symptom

Tests, builds, and deploys ran through agent calls. That meant token spend on work with a deterministic, scriptable answer — and hallucination risk on tasks where a wrong answer causes real damage (a broken deploy, a skipped test).

AGENT task: "run npm test" # → full LLM call # → token cost # → hallucination risk # → non-deterministic output
TEO 5 — The fix

A SCRIPT task is the default. The orchestrator runs it directly at zero tokens. Sage's litmus test: "Could a human do this by running a fixed command?" If yes → SCRIPT. LLM only when judgment is genuinely required.

SCRIPT task: "run npm test" # → orchestrator executes directly # → 0 tokens # → deterministic exit code # AGENT only when judgment needed
06

Unattributable verdict

v4.x — The symptom

A false positive traced back to an ambient token — not a person, not a specific agent. There was no way to answer "who signed this?" The audit trail stopped at the token file, which had no signer field.

verdict: PASS signer: ??? ← no actor_id source: ambient token file # impossible to attribute
TEO 5 — The fix

Every event names exactly one actor_id. A false positive traces to one signer — human or agent. The HMAC signature is over a payload that includes the actor_id, so the attribution is part of what's verified, not an annotation added after the fact.

verdict: PASS actor_id: sage/plan:abc123 sig: HMAC-SHA256(...) # actor_id inside the signed payload
