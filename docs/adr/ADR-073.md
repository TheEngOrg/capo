# ADR-073: src/ Is the Only Source of Truth ŌĆö plugin/ Is a Build Artifact

## Status

Accepted

## Context

Before v1.1.0, the repo maintained two parallel trees of plugin content ŌĆö authored source under `src/agents/hooks/skills/` and a second committed copy in `agents/hooks/skills/` (or `plugin/`). This caused the v1.0.0 production incident: `agents/capo.md` diverged from its source counterpart and shipped the stale relay model. Having the built artifact on `main` alongside authored source made drift structurally inevitable.

A second issue: the `agents` field in `plugin.json`. The Claude Code plugin loader has three behaviors:

| `agents` field value | Loader behavior |
|---|---|
| Explicit file array | Agents(0) silently ŌĆö no error, zero agents loaded |
| Directory string | Install fails |
| Field omitted entirely | Auto-discovers all `*.md` in the agents directory ŌĆö works |

`claude plugin validate` passes all three. The real gate is `claude plugin install` + `claude plugin details`.

## Decision

**`src/` is the only source of truth for all authored content, including agents, hooks, and skills.**

All plugin content lives under `src/plugin/agents|hooks|skills`. `plugin/` is a build artifact: gitignored, never committed to `main`, never hand-authored.

**The build step** (`scripts/build-plugin.mjs`, run via `npm run build:plugin`) copies `src/plugin/` into a clean `plugin/` directory, strips the `agents` field from `plugin.json` (triggering auto-discovery), syncs version from `package.json`, and rewrites paths to be relative to the plugin root.

**The `release` branch** holds the built plugin and nothing else. Marketplace `path: "."`, floating `ref: "release"`. Users get updates via `claude plugin update` with no re-pin.

**The invented engine agent schema is retired.** Old `src/agents/*.md` used `agent_id`/`role`/`disallowedTools_default` fields that the Claude Code runtime never parsed. One roster only: Claude Code format (`name`, `model`, `tools`).

npm script changes: `build:plugin` and `build:all` added; `postinstall` removed; `bundle` retained for the engine binary only.

## Consequences

**Positive:**
- Drift between authored source and deployed plugin is structurally impossible on `main`.
- `plugin/` absent from the working tree eliminates the context confusion that caused the v1.0.0 incident.
- The `agents` field is stripped deterministically at build time ŌĆö cannot accidentally re-introduce Agents(0).
- One roster format removes the dual-schema maintenance burden.

**Negative:**
- Devs must run `npm run build:plugin` before `npm run dev:install`. No hot-reload.
- The `release` branch is a separate deployment surface that `release.sh` must update on every release.
- `plugin/` being gitignored means CI cannot verify the built artifact. `verify-plugin-install.sh` (run by Brodie) is the only trustworthy acceptance gate.

**Open questions:**
- Should CI run `npm run build:plugin` and assert the output is non-empty as a smoke check?
- Should a CI job validate that `release` branch tip matches the latest `main` tag to catch a partial `release.sh` run?
