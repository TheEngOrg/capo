# ADR-074: UserPromptSubmit for Per-Prompt Skill Activation

## Status

Accepted

## Context

The initial CAPO plugin used `SessionStart` (`capo-activation.sh`) to inject a version banner and Capo activation signal once at session open. The assumption was that one-time context injection would be sufficient for Claude to recognize `/teo` prompts throughout the session.

This assumption was wrong. Claude is goal-focused — after session start context is loaded, Claude shifts attention to the user's task. Routing reminders from `SessionStart` don't stay active in the model's attention window when a `/teo` prompt arrives 10 or 30 turns in. The symptom: `/teo` prompts triggered Claude's default behavior instead of routing to `teo:capo`.

The hook event contract has two relevant options:

| Event | Fires | `additionalContext` | `decision: "block"` | Timeout |
|---|---|---|---|---|
| `SessionStart` | Once at session open | Yes | No | 600s |
| `UserPromptSubmit` | Before every user prompt | Yes | Yes | 30s |

`SessionStart` is right for session-scoped init. `UserPromptSubmit` fires at exactly the moment routing needs to happen.

## Decision

**`UserPromptSubmit` is the canonical hook event for per-prompt skill routing in the CAPO plugin.**

Implementation: `teo-prompt-router.sh`, registered under `UserPromptSubmit` in `hooks.json`.

Routing logic:
1. Extract `.prompt` from stdin JSON via `jq`. Missing `jq` → exit 0 (fail-open).
2. Non-`/teo` prompts → exit 0 immediately. Zero overhead.
3. Bare `/teo` or utility keywords (`help`, `status`, `version`, `list`, `stop`) → exit 0. Skill handles these.
4. Substantive `/teo` prompts → emit JSON with `hookSpecificOutput.additionalContext: "CAPO_DIRECTIVE: route this prompt to teo:capo for orchestration"`.

The hook always exits 0 — never blocks. `decision: "block"` is deliberately unused for routing; a mis-classification should fail open rather than silently drop the user's prompt.

**`SessionStart` is retained but narrowed** to version banner and provisioning signals only — no routing or skill-activation context.

Plain stdout from `UserPromptSubmit` is also added as context (fallback for simpler scripts). The JSON `additionalContext` path is preferred.

## Consequences

**Positive:**
- `/teo` prompts reliably activate Capo routing anywhere in a long session.
- Non-`/teo` prompts exit in microseconds. No per-prompt overhead for non-TEO users.
- `decision: "block"` capability is available for future enforcement use cases without a hook architecture change.
- Pure shell, no Node.js dependency, no filesystem or network I/O — well within the 30s timeout.

**Negative:**
- 30s `UserPromptSubmit` timeout is much tighter than `SessionStart`'s 600s. Future additions must be explicitly bounded.
- Without `jq`, routing degrades silently. The fail-open is deliberate but means the guarantee is conditional on `jq` availability.
- `additionalContext` is a strong signal, not an enforcement constraint. A sufficiently long context or contradicting earlier instruction could still cause Claude to ignore it.

**Open questions:**
- Should the README note `jq` as an explicit install dependency?
- The `TEO_VALIDATE_ARTIFACT=1` env stub in `teo-prompt-router.sh` has no tests and no wiring. Wire it when ADR-072 WS-00 lands the artifact schema layer, or remove it before it becomes dead code.
